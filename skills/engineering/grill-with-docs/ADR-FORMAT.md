# ADR 格式

ADR 存放在 `docs/adr/`，使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

懒创建 `docs/adr/` 目录——只在第一份 ADR 需要时创建。

## 模板

```md
# {Short title of the decision}

{1-3 sentences: what's the context, what did we decide, and why.}
```

就这么多。一份 ADR 可以只有一段。价值在于记录**有过一个决定**以及**为什么**——而不是把分节填满。

## 可选小节

只在它们真正带来价值时才包含。多数 ADR 不需要。

- **Status** 前置数据（`proposed | accepted | deprecated | superseded by ADR-NNNN`）— 决定被重新审视时有用
- **Considered Options** — 仅当被拒绝的备选方案值得记下来时
- **Consequences** — 仅当需要点出非显而易见的下游影响时

## 编号

扫描 `docs/adr/` 中最高的现有编号，加 1。

## 何时建议写 ADR

以下三条必须全部为真：

1. **难以回退** — 后续改主意的代价不小
2. **没有上下文会令人意外** — 未来的读者看了代码会想"为什么他们这么做？"
3. **真正取舍的结果** — 存在真正的备选项，你为特定原因选了一个

如果一个决定容易回退，跳过——你回头就能改。如果它不令人意外，没人会想为什么。如果根本没有真正的备选，那除了"我们做了显而易见的事"以外没什么可记。

### 哪些适合

- **架构形态。** "我们用 monorepo。" "写模型是事件溯源的，读模型投射到 Postgres。"
- **不同上下文之间的集成模式。** "Ordering 与 Billing 通过领域事件通信，不用同步 HTTP。"
- **带锁定的技术选型。** 数据库、消息总线、认证提供方、部署目标。不是每个库——只有那些要花一整季才能换掉的。
- **边界和范围决定。** "客户数据归 Customer 上下文所有；其他上下文只通过 ID 引用。" 显式的"不"和"是"一样有价值。
- **故意偏离显而易见的路径。** "我们用手写 SQL 而不是 ORM 是因为 X。" 任何合理读者会假设相反做法的地方。这能阻止下一个工程师"修复"那些本来就是故意为之的东西。
- **代码中看不到的约束。** "我们不能用 AWS 因为合规要求。" "响应时间必须低于 200ms 因为合作方 API 的合约。"
- **拒绝理由非显而易见的备选。** 如果你考虑过 GraphQL，因微妙原因选了 REST，记下来——否则六个月后会有人再次建议 GraphQL。
