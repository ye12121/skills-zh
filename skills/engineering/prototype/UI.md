# UI 原型

在同一路由上生成 **几种大相径庭的 UI 变体**，通过悬浮底栏切换。用户在浏览器里翻看变体，选一个（或从每个里挑零件），然后把其余的扔掉。

如果问题是关于逻辑/状态而不是外观——选错分支了。用 [LOGIC.md](LOGIC.md)。

## 何时是合适的形状

- "这页应该长什么样？"
- "我想在承诺之前看几个 dashboard 的方案。"
- "为设置页试一个不一样的布局。"
- 任何用户原本会花一天在脑中三个含糊草图之间纠结的场景。

## 两种子形态——强烈优先选择子形态 A

UI 原型在**与应用其余部分相邻**时更容易判断——真实 header、真实侧栏、真实数据、真实密度。孤立的一次性路由是真空：每个变体单独看都还行。只要存在合理的现有页面可以宿主变体，就默认选子形态 A。只有在原型确实没有附近落脚处时才考虑子形态 B。

### 子形态 A — 在现有页面上调整（首选）

路由已存在。变体渲染在**同一路由**上，由 `?variant=` URL search param 控制。现有的数据获取、参数、鉴权都保留——只切换渲染。这是默认；没特殊原因就选它。

如果原型针对的东西还没有页面但**自然会被嵌入某个页面**（dashboard 的新区段、设置屏上的新卡片、现有流程中的新步骤）——这也是子形态 A。把变体挂载到宿主页面里。

### 子形态 B — 新页面（不得已）

只在被原型化的东西确实没有现有页面可以落脚时使用——例如全新的顶层入口，或无法嵌入任何合理位置的流程。

按项目现有路由约定创建**一次性路由**——不要发明新的顶层结构。命名要让它显然是原型（例如在路径或文件名里包含 `prototype`）。同样的 `?variant=` 模式。

承诺子形态 B 之前合理性检查：真的没有现有页面可以嵌进去吗？空路由会掩盖那些有内容的页面会暴露的设计问题。

两种子形态下，悬浮底栏完全相同。

## 流程

### 1. 陈述问题并选 N

默认 **3 个变体**。超过 5 个就不再是大相径庭而变成噪音——上限就是这里。

把计划写成一句话，放在原型所在位置或文件顶部注释里：

> "Three variants of the settings page, switchable via `?variant=`, on the existing `/settings` route."

无论用户在场还是不在，这都成立。

### 2. 生成大相径庭的变体

起草每个变体。每个都对照：

- 该页的目的及其可用的数据。
- 项目的组件库 / 样式系统（TailwindCSS、shadcn、MUI、纯 CSS，无论是什么）。
- 一个清晰的导出组件名，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须 **在结构上不同**——不同的布局、不同的信息层级、不同的主交互可见点，而不是只换颜色。三个细微调整过的卡片网格不是 UI 原型，是壁纸。如果两个草稿出来太像，明确给出"不要使用卡片网格"的指导重做其中一个。

### 3. 把它们连起来

在路由上创建一个切换组件：

```tsx
// pseudo-code — adapt to the project's framework
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形态 A（现有页面）：所有现有数据获取保留在 switcher 之上；只有渲染子树按变体变化。

对于子形态 B（新页面）：`/prototype/<name>` 下的一次性路由挂载同一个 switcher。

### 4. 构建悬浮 switcher

屏幕底部居中的小型固定位置栏，三块：

- **左箭头** — 切到上一个变体（循环）。
- **变体标签** — 显示当前变体键，如果变体导出名字则也显示，例如 `B — Sidebar layout`。
- **右箭头** — 向前切（循环）。

行为：

- 点击箭头更新 URL search param（用框架的 router——Next 上 `router.replace`，React Router 上 `navigate` 等），让变体可分享且重载稳定。
- 键盘：`←` 和 `→` 也切换。当 `<input>`、`<textarea>` 或 `[contenteditable]` 获得焦点时不要拦截方向键。
- 视觉上与页面区分（例如高对比胶囊、微妙阴影）——明显不是被评估设计的一部分。
- 在生产构建中隐藏——用 `process.env.NODE_ENV !== 'production'` 或等效检查门控，避免一次失手合并的原型把这个栏推到用户面前。

把 switcher 放进一个共享组件，两种子形态都能复用。放在项目里共享 UI 所在的位置。

### 5. 交付

把 URL（和各 `?variant=` 键）告知用户。他们方便时翻看。最有意思的反馈通常是 **"我想要 B 的 header 配 C 的侧栏"**——那才是他们真正想要的设计。

### 6. 捕获答案并清理

一旦某个变体胜出，写下是哪个以及为什么（提交信息、ADR、Issue，或者紧挨原型的 `NOTES.md`，如果是 AFK 跑的且用户尚未回复）。然后：

- **子形态 A** — 删除落选变体和 switcher；把胜出者折回到现有页面。
- **子形态 B** — 把胜出变体提升为真实路由，删掉一次性路由和 switcher。

不要把变体组件或 switcher 留在那。它们烂得快，会让下一个读者困惑。

## 反模式

- **只在颜色或文案上不同的变体。** 那是微调，不是原型。真正的变体在结构上不一致。
- **变体之间共享太多代码。** 共享一个 `<Header>` 没问题；共享一个 `<Layout>` 就违背了目的。每个变体都应当能自由扔掉布局。
- **把变体接到真实变更上。** 只读原型没问题。如果某变体需要写入，指向一个 stub——问题是"这应该长什么样"，不是"后端能不能用"。
- **把原型直接提升到生产。** 变体代码是在原型约束下写的（无测试、最少错误处理）。折回去时正式重写。
